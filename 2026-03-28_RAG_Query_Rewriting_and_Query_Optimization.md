## 1. 머리글

RAG 시스템에서 Retrieval 성능은 문서뿐만 아니라 질의(query)의 형태에도 크게 영향을 받는다. 동일한 의미를 가진 질문이라도 표현 방식에 따라 검색 결과가 달라질 수 있다.

사용자의 질문은 대부분 자연어로 작성되며, 검색에 최적화된 형태가 아닌 경우가 많다. 반면 문서는 기술 문서, 매뉴얼, 규정 등 특정 형식으로 작성되어 있는 경우가 많다. 이 두 표현 방식 사이의 차이가 Retrieval 성능 저하의 원인이 된다.

이 문제를 해결하기 위한 접근이 Query Rewriting이다. Query Rewriting은 질문을 검색에 더 적합한 형태로 변환하여 Retrieval 성능을 개선하는 과정이다.

---

## 2. Query-Document Mismatch

Query-Document Mismatch는 질문과 문서가 서로 다른 표현 공간에 있을 때 발생한다. 예를 들어 사용자는 다음과 같이 질문할 수 있다.

설정을 어떻게 바꾸지?

하지만 문서에는 다음과 같은 표현이 사용될 수 있다.

설정 변경 절차는 다음과 같다.

두 문장은 의미적으로 연결될 수 있지만, embedding 공간에서는 충분히 가깝지 않을 수 있다. 특히 방법과 절차, 어떻게와 조건처럼 표현 방식이 다르면 검색 성능이 크게 떨어질 수 있다.

이 문제는 문서 품질의 문제가 아니라 질의 표현 문제에 해당한다.

---

## 3. Query Rewriting의 기본 개념

Query Rewriting은 사용자의 질문을 그대로 사용하는 것이 아니라, 검색에 더 적합한 형태로 변환하는 과정이다. 이 과정은 다음과 같은 방향으로 이루어진다.

- 질문을 명확한 키워드 중심 표현으로 변환  
- 불필요한 수식어 제거  
- 질문 의도를 명시적으로 드러내는 형태로 재구성  

예를 들어 다음과 같은 변환이 가능하다.

원질문: 이거 설정하려면 뭐 해야 해?  
변환 질문: 서비스 설정 변경 방법  

이처럼 질의를 문서 표현 방식에 맞추면 Retrieval 성능이 개선될 수 있다.

---

## 4. LLM 기반 Query Rewriting

최근에는 LLM을 활용하여 Query Rewriting을 수행하는 방식이 많이 사용된다. LLM은 질문의 의도를 파악한 뒤, 이를 다양한 형태로 재구성할 수 있다.

대표적으로 다음과 같은 방식이 있다.

- Query Expansion: 하나의 질문을 여러 형태로 확장  
- Query Reformulation: 질문을 더 명확한 형태로 재작성  
- Step-back Question: 질문을 더 일반적인 형태로 변환  

예를 들어,

원질문: 이 설정 왜 안돼?  
Reformulation: 설정 실패 원인  
Expansion: 설정 오류 원인/설정 실패 조건/설정 제한 사항  

이러한 방식은 embedding 기반 검색에서 놓칠 수 있는 문서를 찾는 데 도움을 준다.

---

## 5. Multi-query Retrieval

Query Rewriting의 확장 형태로 multi-query retrieval이 사용된다. 이 방식에서는 하나의 질문을 여러 개의 질의로 변환한 뒤, 각각의 결과를 결합한다.

구조는 다음과 같다.

① 질문 → 여러 query 생성  
② 각 query로 retrieval 수행  
③ 결과를 병합 및 정렬  

이 방식은 단일 query에서 놓칠 수 있는 문서를 보완할 수 있다. 특히 표현 방식이 다양한 문서 집합에서는 효과가 크다.

다만 query 수가 증가할수록 검색 비용이 증가하기 때문에, 적절한 균형이 필요하다.

---

## 6. Query Rewriting의 한계

Query Rewriting은 Retrieval 성능을 개선하는데 효과적이지만, 몇 가지 한계를 가진다.

첫 번째는 과도한 확장 문제이다. query를 지나치게 많이 생성하면 irrelevant 문서가 함께 검색될 가능성이 높아진다. 이는 over-retrieval로 이어질 수 있다.

두 번째는 의도 왜곡 가능성이다. LLM이 질문을 재작성하는 과정에서 사용자의 원래 의도가 변형될 수 있다.

세 번째는 연산 비용 증가이다. multi-query retrieval이나 LLM 기반 rewriting은 시스템 비용을 증가시키는 요인이 된다.

따라서 Query Rewriting은 무조건 적용하기보다는, Retrieval 성능이 부족한 경우에 선택적으로 사용하는 것이 바람직하다.

---

## 7. 마무리

RAG 시스템에서 Retrieval 성능은 문서뿐 아니라 질의의 표현 방식에도 크게 의존한다. Query-Document Mismatch는 단순한 검색 실패가 아니라, 표현 방식의 차이에서 발생하는 구조적인 문제이다.

Query Rewriting은 이러한 문제를 해결하기 위한 방법으로, 질문을 검색에 더 적합한 형태로 변환함으로써 Retrieval 성능을 개선한다.

하지만 이 과정은 추가적인 비용과 복잡성을 동반하며, 항상 성능 향상으로 이어지는 것은 아니다. 따라서 Query Rewriting은 Retrieval 설계의 일부로서, 다른 요소들과 함께 고려되어야 한다.

결국 RAG에서 중요한 것은 문서를 어떻게 찾느냐뿐만 아니라, 무엇을 질문하느냐와 그 질문을 어떻게 표현하느냐이다.
