## 1. 머리글

RAG 시스템 구축 후 가장 신경 써야 하는 일은 그 성능을 평가하는 일이다. 일반적인 NLP 모델과 달리 RAG 시스템은 Retrieval과 Generation 두 단계로 구성되기 때문에 단일 지표로 전체 성능을 판단하기 어렵다.

예를 들어 답변이 틀렸다고 해서 항상 모델이 잘못된 것은 아니다. Retrieval 단계에서 필요한 문서를 찾지 못했다면 생성 단계는 애초에 올바른 답을 만들 수 없다. 반대로 Retrieval이 성공했음에도 잘못된 답변이 생성되었다면 이는 Generation 단계의 문제일 가능성이 높다.

따라서 RAG 시스템의 평가에서는 Retrieval과 Generation을 분리해서 살펴보는 것이 중요하다. 두 단계의 오류를 구분하지 않으면, 실제 문제와 다른 방향으로 시스템을 수정하게 될 가능성이 높다.

---

## 2. RAG 평가의 기본 구조

RAG 시스템 평가는 일반적으로 다음 세 단계로 나누어 생각할 수 있다.

| 평가 단계 | 평가 대상 | 주요 질문 |
|---|---|---|
| Retrieval Evaluation | 검색된 문서 | 필요한 정보를 찾았는가 |
| Generation Evaluation | 생성된 답변 | 질문에 올바르게 답했는가 |
| Faithfulness Evaluation | 답변과 문서 관계 | 답변이 문서에 의해 지지되는가 |

이 세 가지 평가는 서로 독립적인 것이 아니라, 연쇄적인 관계를 가진다. Retrieval이 실패하면 Generation 평가 자체가 의미 없어질 수 있으며, Generation이 성공했더라도 Faithfulness 문제가 남을 수 있다.

따라서 RAG 평가의 핵심은 단순히 답변 정확도를 보는 것이 아니라, 어떤 단계에서 오류가 발생했는지를 구분하는 것이다.

---

## 3. Retrieval 평가

Retrieval 평가는 검색 시스템이 질문에 필요한 문서를 얼마나 잘 찾아오는지를 측정한다. 이 단계에서는 LLM의 생성 능력은 고려하지 않고, 검색 결과 자체의 품질만을 평가한다.

대표적으로 사용되는 지표는 다음과 같다.

| 지표 | 의미 |
|---|---|
| Recall@k | 상위 k개 문서 안에 정답 문서가 포함되는 비율 |
| Precision@k | 상위 k개 문서 중 실제 관련 문서의 비율 |
| MRR (Mean Reciprocal Rank) | 정답 문서의 평균 순위 |

예를 들어 Recall@5가 0.8이라는 것은, 질문의 80%에서 정답 문서가 Top-5 검색 결과 안에 포함되었다는 의미이다.

Retrieval 평가는 RAG 시스템에서 매우 중요한 단계이다. Retrieval이 실패하면 이후 단계에서 아무리 좋은 LLM을 사용하더라도 올바른 답변을 생성하기 어렵기 때문이다.

---

## 4. Generation 평가

Generation 평가는 모델이 생성한 답변 자체의 품질을 평가하는 단계이다. 이때는 검색된 문서가 아니라 최종 출력 텍스트를 기준으로 평가한다. 전통적인 NLP에서는 다음과 같은 지표들이 자주 사용된다.

| 지표 | 특징 |
|---|---|
| BLEU | n-gram 기반 문장 유사도 |
| ROUGE | 요약 모델 평가에 많이 사용 |
| METEOR | 의미 기반 유사도 고려 |

하지만 이러한 지표는 RAG 평가에 항상 적합한 것은 아니다. 동일한 질문에 대해 여러 표현의 정답이 존재할 수 있기 때문이다.

예를 들어 다음 두 문장은 의미적으로 동일하지만 n-gram 기준에서는 낮은 점수를 받을 수 있다.

설정을 완료하려면 관리자 권한이 필요합니다.  
이 작업은 관리자 권한으로 실행해야 합니다.

이 때문에 최근에는 LLM을 활용한 LLM-as-a-judge 방식의 평가도 많이 사용된다.

---

## 5. Faithfulness 평가

Generation 평가만으로는 RAG 시스템의 품질을 충분히 설명할 수 없다. 답변이 맞더라도 문서에 근거하지 않은 경우가 존재하기 때문이다.

Faithfulness 평가는 답변이 제공된 문서에 의해 지지되는지를 확인하는 과정이다. 이는 RAG 시스템에서 Hallucination을 측정하는 가장 직접적인 방법이다. 예를 들어 다음과 같은 상황을 생각할 수 있다.

문서: 서비스 이용에는 이메일 인증이 필요하다.  
모델 답변: 서비스 이용에는 이메일 인증과 전화번호 인증이 필요하다.

이 답변은 부분적으로 맞을 수 있지만, 문서에 없는 정보가 포함되어 있기 때문에 Faithfulness 기준에서는 실패로 판단된다. Faithfulness 평가는 다음과 같은 방식으로 수행된다.

- 문서와 답변을 함께 모델에 입력하여 근거 여부 판단
- 사람이 직접 문서 기반 여부를 검증
- claim 단위로 문장을 분해하여 평가

이 과정은 단순 정확도 평가보다 비용이 높지만, RAG 시스템의 신뢰성을 판단하는 데 매우 중요한 지표이다.

## 6. RAG 평가에서 자주 발생하는 문제

RAG 평가에서 가장 흔히 발생하는 실수는 Retrieval과 Generation 오류를 구분하지 않는 것이다. 예를 들어 다음과 같은 상황이 있을 수 있다.

- Retrieval 실패 → 모델이 추측으로 답변 생성
- Generation 실패 → 문서는 있지만 답변을 잘못 생성

두 경우 모두 '답변이 틀렸다'는 결과만 보면 동일하게 보인다. 하지만 실제 원인은 전혀 다르며, 해결 방법도 다르다. 따라서 RAG 시스템을 평가할 때는 다음과 같은 질문을 순서대로 확인해야 한다.

- 필요한 문서가 검색되었는가
- 문서를 기반으로 답변이 생성되었는가
- 답변이 문서에 의해 지지되는가

이 과정을 거쳐야만 시스템 개선 방향을 정확히 판단할 수 있다.

---

## 7. 마무리

RAG 시스템의 평가는 단순한 정확도 측정으로 해결되지 않는다. Retrieval, Generation, Faithfulness라는 서로 다른 단계가 존재하기 때문이다.

특히 RAG에서는 Retrieval 오류와 Generation 오류가 서로 다른 방식으로 나타난다. 이를 구분하지 않으면 시스템 개선 과정에서 잘못된 판단을 내릴 가능성이 높다.

따라서 RAG 평가의 핵심은 정답 여부를 확인하는 것이 아니라, 오류의 위치를 찾는 것이라고 볼 수 있다. Retrieval 품질이 충분히 확보된 이후에야 Generation 개선이 의미를 가지며, 그 이후에 Fatihfulness 문제가 논의될 수 있다.
