# 1. 머리말

RAG는 단일 모델이 아니라 두 개의 서로 다른 시스템을 결합한 구조이다. 이를 구성하는 시스템은 질의와 관련 정보를 찾는 **Retrieval 시스템**, 주어진 입력을 바탕으로 답변을 생성하는 **Generation 시스템(LLM)**이다. 이 둘은 순차적으로 연결되며, 생성 단계가 검색 결과에 조건부로 의존하게끔 설계되어 있다.

RAG는 검색으로 입력을 확장한 뒤 그 입력을 조건으로 생성되는데, 여기서 중요한 점은 **검색이 생성 기능의 보조 기능이 아니라 생성의 전제 조건 그 자체**라는 것이다.

---

# 2. 기본적인 RAG 파이프라인

RAG의 파이프라인은 기본적으로 다음 4단계로 구성된다.

1. **Query Interpretation**  
   사용자 질의를 그대로 쓰거나, 검색에 적합한 형태로 변환한다.

2. **Retrieval**  
   임베딩 기반 검색을 통해 관련 문서 조각(chunks)을 Top-k로 가져온다.

3. **Context Construction**  
   검색 결과를 정제·정렬하여 LLM 입력 컨텍스트로 구성한다.

4. **Generation**  
   LLM이 질의와 컨텍스트를 함께 조건으로 답변을 생성한다.

이 흐름에서 검색과 생성은 독립적이지 않다.  
**검색 품질은 생성 품질의 상한을 결정**하며, 그렇기에 생성 단계의 실패 원인은 대부분 이전 단계인 검색 단계에서 비롯된다.

---

# 3. 조건부 분포의 확장

RAG 구조를 확률적 관점에서 보면, 생성 품질이 검색 품질에 크게 의존한다는 사실이 더욱 분명해진다.

LLM 단독 시스템은 다음 확률을 근사한다.

- P(y | x)

반면 RAG는 검색된 문서 `d`를 조건에 포함시킨다.

- P(y | x, d₁, d₂, ..., dₖ)

이러한 구조에서는 LLM이 부족한 입력을 스스로 보완하려고 할 필요가 줄어든다.  
입력이 불충분한 경우에도, 모델은 **주어진 문서에 기반해 답변을 구성하도록 유도**된다.

결과적으로 Hallucination을 완전히 제거할 수는 없지만, **발생 가능한 공간을 제한하는 효과**를 가진다.

---

# 4. 기초 RAG가 작동하는 이유 및 구조적 한계

기초 RAG(Naive RAG)가 효과적으로 작동하는 이유는 비교적 명확하다. 대부분의 질의는 복잡한 추론보다 **관련 정보의 존재 여부**가 성능을 더 크게 좌우하며, 검색 결과가 충분히 관련 있다면 LLM은 이를 자연스럽게 요약·재구성해 안정적인 답변을 생성할 수 있다.

또한 RAG는 Fine-tuning과 달리 학습 비용이 없고, 문서 업데이트가 즉시 반영되므로 **최신성 유지가 용이**하다. 이러한 이유로 기초 RAG만으로도 실무에서 충분한 성과를 내는 경우가 많다.

그러나 같은 이유로 기초 RAG는 쉽게 실패하기도 한다. 구조적인 관점에서 보면 다음과 같은 한계를 가진다.

- 검색 품질에 전적으로 의존한다.  
- 검색된 문서의 관련성을 LLM이 스스로 판단해야 한다.  
- 불필요한 문서가 많을수록 생성 품질이 오히려 하락한다.  
- 검색 실패와 생성 실패가 출력 단계에서 혼재된다.  

따라서 RAG의 실패는 구조적인 문제가 아니라, **설계의 부족**에서 발생한다.

---

# 5. 검색 오류와 생성 오류의 구분

RAG 시스템의 오류를 해결하기 위해서는, 해당 오류가 **검색 오류인지 생성 오류인지**를 먼저 구분할 수 있어야 한다.

LLM이 부정확한 답변을 생성했다고 해서, 문제가 반드시 생성 단계에 있다고 단정할 수는 없다. 실제로는 검색 실패나 필수 정보 누락이 원인인 경우도 많다.

RAG의 중요한 장점은 **실패 원인이 어느 단계에 귀속되는지 비교적 명확히 파악할 수 있다는 점**이다.  
따라서 오류 분석 시 다음 중 어디에 문제가 있었는지를 판단해야 한다.

- Query 처리  
- Retrieval  
- Context 구성  
- Generation  

이는 RAG가 **디버깅 가능한 구조**임을 보여준다.

---

# 6. 마무리

RAG의 기본 구조는 검색을 통해 입력을 확장하고, 생성이 그 입력에 의존하도록 만드는 매우 단순한 설계이다. 그러나 이 설계를 어떻게 다루느냐에 따라, 파라미터 변경 없이도 RAG의 성능은 크게 달라질 수 있다.

엔지니어는 다음과 같은 설계 요소를 고려해야 한다.

- Query를 그대로 사용할 것인지, 변환할 것인지  
- Top-k를 몇 개로 설정할 것인지  
- 문서를 어떤 순서와 형식으로 입력할 것인지  
- 검색 실패 시 어떻게 대응할 것인지  

보다 효과적인 RAG 설계를 위해 이러한 선택들을 신중히 다루는 것이, LLM 오류를 줄이기 위한 핵심 과제가 된다.
